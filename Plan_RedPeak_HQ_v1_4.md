# 1. Назначение системы

RedPeak HQ — операционная система руководителя проекта RedPeak (ООО «РЭДПИК»). Цифровой штаб: память + архив + диспетчеризация + подготовка решений.

Система решает главную проблему руководителя-основателя: необходимость одновременно быть TeamLead, юристом, бухгалтером, плановиком при команде из 4 человек. Главный bottleneck — чтение, удержание контекста, сведение информации и подготовка решения.

**Единый цикл работы:**

- Руководитель даёт задачу в свободной форме

- Диспетчер (CLAUDE.md) проводит intake-опрос: цель, контур, срок, форма результата

- Задача маршрутизируется нужному субагенту

- Субагент выполняет задачу, используя файлы и память

- Субагент проводит self-check результата перед выдачей

- Результат выдаётся в запрошенном формате

- Значимые факты записываются в память (только с подтверждением)

### strategy-agent: детализация роли

**Назначение:** Мета-агент, который видит проект целиком — поверх данных всех остальных субагентов — и даёт руководителю точечные, обоснованные рекомендации. Не заменяет принятие решений, а готовит их.

**Входные данные (собирает из памяти и файлов):**

- Текущие приоритеты и дедлайны (operations-agent)
- Открытые обязательства и риски по договорам (legal-agent)
- Бюджет, burn rate, остаток гранта (accounting-agent)
- Вехи 14-недельного плана разработки
- Статусы по ФСИ / Сколково
- Calibration log (ошибки и паттерны системы)

**Режимы работы:**

| **Режим** | **Триггер** | **Выход** |
| --- | --- | --- |
| **Еженедельный брифинг** | Автоматически (первый понедельничный чат) или по запросу | Стратегическая сводка: 3–5 точечных рекомендаций с обоснованием |
| **Анализ развилки** | Руководитель описывает ситуацию выбора | Таблица вариантов: аргументы за/против, риски, рекомендация |
| **Проактивный сигнал** | При холодном старте: если обнаружены критические паттерны | Алерт: «Обрати внимание: [факт] → [риск] → [рекомендуемое действие]» |

**Формат еженедельной стратегической сводки:**

1. **Фокус недели** — одна главная задача, обоснование почему именно она
2. **Точечные рекомендации** (3–5 шт.) — конкретное действие + почему сейчас + что будет, если отложить
3. **Упущенные возможности** — что можно сделать, но руководитель, вероятно, не помнит/не видит
4. **Ранние сигналы рисков** — что ещё не проблема, но станет ей через 1–2 недели
5. **Вопрос руководителю** — один вопрос, ответ на который разблокирует следующие действия

**Правила strategy-agent:**

- Каждая рекомендация привязана к конкретному факту из памяти или файла. Нет факта — нет рекомендации.
- Не даёт абстрактных советов типа «стоит подумать о масштабировании». Только конкретика: кто, что, когда, зачем.
- При анализе развилки — всегда минимум 2 варианта. Если рекомендует один — объясняет, почему не другой.
- Маркирует уровень уверенности: [ВЫСОКАЯ] — из документов, [СРЕДНЯЯ] — из анализа паттернов, [ГИПОТЕЗА] — требует проверки.
- Не дублирует работу других агентов. Не пересказывает статусы. Только стратегический слой поверх.

# 3. Защита от галлюцинаций

## 3.1. Правила в CLAUDE.md (диспетчер)

- НИКОГДА не выдумывай факты. Если не знаешь — скажи «я не знаю» или «требует проверки».

- НИКОГДА не выдумывай ссылки на ГОСТы, статьи законов, номера документов, даты, суммы.

- Чётко разделяй: (1) из документов проекта, (2) моя интерпретация, (3) требует внешней проверки.

- При утверждении фактов — указывай источник: конкретный файл или запись в памяти.

- При работе с числами, датами, суммами — ВСЕГДА цитируй конкретный файл-источник. Нет файла — помечай [ДАННЫЕ ОТСУТСТВУЮТ].

- При обсуждении рынка, законодательства, текущих событий — используй веб-поиск.

## 3.2. Ground truth: привязка к файлам-источникам

Каждый факт в результате работы агента должен быть привязан к конкретному источнику. Это «ground truth» — проверяемая опора.

| **Тип данных** | **Источник ground truth** | **Если источника нет** |
| --- | --- | --- |
| Сумма, цена | Файл из documents/finance/ или documents/contracts/ | [ДАННЫЕ ОТСУТСТВУЮТ] |
| Дата, дедлайн | Файл договора, запись в памяти, документ ФСИ | [ТРЕБУЕТ ПРОВЕРКИ] |
| Номер документа | Файл документа, ЕГРЮЛ, реестр | Не указывать номер вообще |
| Норма закона / ГОСТ | Веб-поиск или загруженный текст ГОСТа | [ПРОВЕРИТЬ НОРМУ] |
| Статус задачи | Запись в памяти, operations/ | Спросить руководителя |

## 3.3. Self-check: evaluator-паттерн Anthropic

Каждый субагент ОБЯЗАН провести самопроверку результата перед выдачей руководителю. Self-check — это не отдельный агент, а обязательный последний шаг в промпте каждого субагента.

**Чек-лист self-check (общий для всех субагентов):**

- Есть ли в результате суммы/даты/номера без указания файла-источника? → Добавить источник или пометить маркером.

- Есть ли ссылки на статьи законов/ГОСТов, которые не проверены через веб-поиск или документ? → Пометить [ПРОВЕРИТЬ НОРМУ].

- Есть ли утверждения, основанные на «понимании контекста», а не на конкретном файле? → Пометить [МОЯ ИНТЕРПРЕТАЦИЯ].

- Противоречит ли результат записям в памяти? → Показать конфликт.

- Соответствует ли формат запрошенному руководителем? → Исправить.

**Дополнительные проверки по ролям:**

| **Субагент** | **Специфичный self-check** |
| --- | --- |
| **legal-agent** | Все пункты договора, упомянутые в анализе, реально существуют в документе? Сверить номера пунктов. |
| **accounting-agent** | Сумма расходов = сумме по позициям? Арифметическая проверка. Статьи бюджета совпадают с утверждённой сметой? |
| **tax-agent** | Сроки подачи деклараций проверены через веб-поиск (не из общих знаний)? |
| **operations-agent** | Дедлайны в плане совпадают с дедлайнами в памяти и в договоре? |
| **strategy-agent** | Каждая рекомендация привязана к конкретному факту (файл/память)? Нет рекомендаций «из воздуха»? Рекомендации не противоречат друг другу? Не дублирует вывод другого субагента? |

## 3.4. Маркеры неуверенности

| **Маркер** | **Когда использовать** |
| --- | --- |
| **[ТРЕБУЕТ ПРОВЕРКИ]** | Факт из общих знаний модели, не из документов проекта |
| **[ПРОВЕРИТЬ НОРМУ]** | Ссылка на закон/ГОСТ без верификации через поиск или документ |
| **[ПРИБЛИЗИТЕЛЬНО]** | Число рассчитано, но входные данные могут быть неполными |
| **[МОЯ ИНТЕРПРЕТАЦИЯ]** | Вывод на основе анализа, а не прямой цитаты |
| **[ДАННЫЕ ОТСУТСТВУЮТ]** | В файлах и памяти нет информации для ответа |

## 3.5. Валидация записей в память

- Факт подтверждён документом или прямым указанием руководителя?

- Если нет — запись помечается тегом needs_review.

- Запись содержит источник (файл, дата обсуждения)?

- Запись не противоречит существующим записям? Если противоречит — показать конфликт.

# 4. Stopping conditions

Субагент может «залипнуть» на сложной задаче, потратив весь контекст. Для предотвращения — обязательные условия остановки.

**Правила для каждого субагента:**

- Если задача требует более 5 обращений к файлам/памяти — остановись, покажи промежуточный результат, спроси руководителя: «продолжать ли?»

- Если при self-check обнаружено более 3 маркеров [ДАННЫЕ ОТСУТСТВУЮТ] — остановись, покажи список недостающих данных, спроси руководителя.

- Если задача не может быть выполнена из-за отсутствия файлов — не выдумывай, покажи что именно нужно и где это должно лежать.

- Максимальная длина одной сессии для сложной задачи — 30 сообщений. После — промежуточное сохранение в файл и предложение начать новый чат.

# 5. Защита при компактировании контекста

## 5.1. Compact Instructions в CLAUDE.md

Секция ## Compact Instructions указывает Claude, что ОБЯЗАТЕЛЬНО сохранять при auto-compact:

- Текущая задача: что делаем, для кого, в каком формате результат

- Активный субагент: какая роль сейчас работает

- Числовые данные: суммы, даты, сроки, номера документов

- Стоп-точки: полный список действий, требующих подтверждения

- Незавершённые шаги: что сделано, что осталось

- Контекст: имя руководителя, проект, команда

## 5.2. Правило коротких сессий

Одна задача = один чат. Не вести в одном чате юридический вопрос, а потом налоговый.

Если задача длинная — каждые 15–20 сообщений сохранять промежуточный результат в файл и записывать прогресс в память. При потере контекста — восстановление из файла и памяти.

## 5.3. Сохранение данных ДО завершения сессии

Числа, даты, суммы, формулы — записать в файл ДО закрытия чата. Контекстное окно ненадёжно. Память (MCP или файлы) — единственное надёжное хранилище.

# 6. Умная память

## 6.1. Фильтрация на входе

| **Записываем** | **НЕ записываем** |
| --- | --- |
| Решения (что, когда, почему). Обязательства (сроки, суммы). Новые факты (контакты, даты). Изменения статуса. Риски. | Промежуточные рассуждения. Отвергнутые варианты. Стилистические правки. Рутинные операции. Неподтверждённые гипотезы. Паспортные данные, реквизиты, пароли. |

## 6.2. Целевые запросы к памяти

Агент НЕ загружает всю базу memory. Вместо этого — целевые запросы:

- «Найди entities типа obligation со статусом актуально» — для проверки дедлайнов

- «Найди entities типа decision, связанные с [тема]» — для контекста решения

- «Найди entities типа risk» — для risk log

Это предотвращает засорение контекста нерелевантной информацией при росте базы.

## 6.3. Формат карточки памяти

Типы entities: decision, obligation, fact, status_change, contact, risk

Обязательные поля: тип, дата, суть, источник, срок действия (если есть), статус (актуально / архив / needs_review).

Каждая observation самодостаточна: через 3 месяца понятно, что решено и почему.

## 6.4. Бэкап и архивация

Ежедневный бэкап: memory/knowledge.jsonl → memory/backups/knowledge_YYYYMMDD.jsonl (PowerShell-скрипт).

Ежеквартальная архивация: записи старше 3 мес. со статусом «архив» → memory/archive/.

## 6.5. Протокол записи

- Субагент формирует карточку записи

- Проверяет: нет ли противоречий с существующими записями

- Помечает needs_review, если факт не из документа

- Показывает карточку руководителю

- Запись — ТОЛЬКО после подтверждения

# 7. Механизм скиллификации

Когда задача третий раз выполняется в похожем формате — диспетчер предлагает создать скилл.

**Структура скилла (progressive disclosure):**

- SKILL.md — точка входа: описание, intake (3–5 вопросов), шаги, правила записи в память. Читается ВСЕГДА.

- TEMPLATE.md — шаблон выхода. Загружается при генерации результата.

- CHECKLIST.md — чек-лист проверки. Загружается при self-check.

- REFERENCE.md — справочные данные (требования ФСИ, нормы, шаблон договора). Загружается только при необходимости.

Вспомогательные файлы НЕ загружаются, пока не нужны — это экономит контекстное окно.

# 8. Механизм самоулучшения системы

Система должна не просто работать, а становиться лучше с каждой неделей — без ручного переписывания промптов руководителем. Самоулучшение основано на замкнутом цикле: выполнение → оценка → обнаружение паттерна → предложение изменения → подтверждение → внедрение.

## 8.1. Цикл обратной связи (Feedback Loop)

Каждая задача проходит полный цикл:

```
Задача → Результат → Оценка → Запись в calibration log → Анализ паттернов → Предложение улучшения → Подтверждение руководителя → Обновление промпта/скилла
```

**Шаг 1. Оценка результата (после каждой задачи):**

Диспетчер спрашивает руководителя (коротко, не навязчиво): «Результат ОК? Если нет — что не так?»

Варианты фиксации:
- ✅ Принят без замечаний
- ⚠️ Принят с правками (руководитель указывает что исправил)
- ❌ Отклонён (руководитель указывает причину)

**Шаг 2. Запись в calibration log:**

Файл: `operations/calibration_log.md`

| **Поле** | **Пример** |
| --- | --- |
| Дата | 2026-03-21 |
| Субагент | legal-agent |
| Задача | Разбор допсоглашения |
| Оценка | ⚠️ правки |
| Проблема | Пропустил пункт 4.3 о неустойке |
| Категория проблемы | missed_data / hallucination / wrong_format / wrong_tone / slow / overkill |
| Действие | Усилить чек-лист contract-review |

**Шаг 3. Анализ паттернов (автоматически, strategy-agent):**

strategy-agent при еженедельном брифинге анализирует calibration_log и ищет повторяющиеся паттерны:

- Один и тот же субагент получает ⚠️/❌ на одном типе задач ≥ 2 раза → **сигнал к изменению промпта**
- Одна и та же категория проблемы повторяется у разных субагентов → **системная проблема** (правило в CLAUDE.md)
- Определённый скилл регулярно требует правок → **скилл нуждается в доработке**

## 8.2. Протокол внесения улучшений

Система НЕ правит себя сама. Но она готовит конкретные предложения.

**Формат предложения улучшения (strategy-agent → руководителю):**

```
ПРЕДЛОЖЕНИЕ УЛУЧШЕНИЯ #[N]

Паттерн: [что повторяется, сколько раз, примеры из calibration_log]
Корневая причина: [почему это происходит — анализ]
Предложение: [конкретное изменение в промпте/скилле/правиле]
Что изменится: [ожидаемый эффект]
Риск: [что может сломаться]
Файл для изменения: [точный путь]

Применить? [Да / Нет / Отложить]
```

**Правила:**

- Любое изменение промпта субагента или правила в CLAUDE.md — ТОЛЬКО после подтверждения руководителя
- Перед изменением — бэкап текущей версии (протокол 15.4)
- После изменения — прогон золотого тест-кейса для затронутого субагента
- Если тест-кейс не пройден — автоматический откат

## 8.3. Уровни самоулучшения

| **Уровень** | **Что улучшается** | **Триггер** | **Кто утверждает** |
| --- | --- | --- | --- |
| L1: Параметрический | Конкретная формулировка в промпте субагента | Паттерн ≥ 2 повторений одной ошибки | Руководитель |
| L2: Структурный | Добавление/изменение правила self-check, нового пункта чек-листа скилла | Паттерн ≥ 3 повторений, категория missed_data или hallucination | Руководитель |
| L3: Архитектурный | Новый скилл, изменение маршрутизации, новое правило в CLAUDE.md | Системная проблема (повторяется у ≥ 2 субагентов) | Руководитель + тест-кейс |
| L4: Скиллификация | Превращение повторяющейся задачи в скилл | Задача выполнена ≥ 3 раз в похожем формате (существующий механизм раздела 7) | Руководитель |

## 8.4. Метрики эволюции системы

strategy-agent ведёт файл `operations/system_evolution.md` — накопительный лог изменений системы.

| **Метрика** | **Как считается** | **Цель** |
| --- | --- | --- |
| Доля ✅ за неделю | ✅ / (✅ + ⚠️ + ❌) | > 80% к концу Фазы 5, > 90% через 2 месяца |
| Среднее кол-во правок на задачу | Сумма правок / кол-во задач | Снижение от недели к неделе |
| Количество применённых улучшений | Счётчик L1–L3 за неделю | Рост в первые 4 недели, затем стабилизация |
| Повторяемость паттернов | Сколько паттернов встретились >2 раз после улучшения | 0 (паттерн исправлен → не повторяется) |
| Регрессии после улучшений | Сколько изменений пришлось откатить | < 10% от общего числа изменений |

## 8.5. Еженедельная ретроспектива (автоматическая)

Каждый понедельник strategy-agent (в рамках еженедельного брифинга) включает блок «Эволюция системы»:

1. **Статистика недели:** сколько задач, распределение ✅/⚠️/❌, по каким субагентам
2. **Обнаруженные паттерны:** новые или повторяющиеся
3. **Предложения улучшений:** готовые к рассмотрению (формат 8.2)
4. **Результаты прошлых улучшений:** что было внедрено, помогло ли (сравнение до/после)
5. **Тренд:** система становится лучше / стагнирует / деградирует

## 8.6. Защитные ограничения самоулучшения

- Система НЕ может ослабить правила антигаллюцинаций (раздел 3). Любое предложение, снижающее проверки — автоматически отклоняется.
- Система НЕ может убрать стоп-точки (раздел 9) или ослабить stopping conditions (раздел 4).
- Система НЕ может изменить правила конфиденциальности (раздел 10).
- Максимум 3 изменения промптов в неделю. Больше — риск каскадных регрессий.
- Каждое изменение живёт минимум 5 задач до следующего изменения того же компонента (период стабилизации).

# 9. Адаптивный тон коммуникации

Система адаптирует язык и стиль под тип задачи. Это не «настройка вежливости», а принципиальное различие: письмо команде и юридический документ — два разных языка.

**Правило для CLAUDE.md и всех субагентов:**

Определи тон ДО генерации текста. Если не очевидно — спроси руководителя: «В каком тоне?»

| **Тип задачи** | **Тон** | **Конкретные правила** |
| --- | --- | --- |
| **Письмо команде** | Живой, человеческий | Короткие предложения. Без длинных тире (—). Без канцелярита. Без «в рамках», «осуществить», «обеспечить реализацию». Как если бы ты писал в мессенджере коллеге, но чуть формальнее. |
| **Письмо партнёру / АСКОН** | Деловой, уважительный | Профессиональная лексика без канцелярита. Конкретика вместо общих фраз. Без подобострастия. |
| **Письмо в ФСИ / Сколково** | Официальный, точный | Формулировки по стандартам отчётности. Ссылки на пункты договора/заявки. Цифры и факты. |
| **Юрид. документ** | Формальный, юридический | Юридическая терминология. Точные формулировки. Нумерация пунктов. Ссылки на нормы. Без разговорных оборотов. |
| **Внутренняя записка** | Прямой, без воды | Суть → обоснование → вывод. Без вступлений. Без «хочется отметить». Таблицы и тезисы вместо абзацев. |
| **Презентация** | Убедительный, структурный | Короткие тезисы. Цифры крупно. Одна мысль на слайд. Без длинных абзацев. |

**Признаки «языка нейронки», которых система ИЗБЕГАЕТ:**

- Длинные тире (—) вместо точек и коротких предложений

- «В рамках реализации проекта», «осуществление деятельности», «обеспечение функционирования»

- Избыточные вводные: «Важно отметить, что...», «Следует подчеркнуть...», «Необходимо учитывать...»

- Одинаковая структура абзацев (тезис → развёрнутое объяснение → вывод) без вариации

- Комплименты задаче: «Отличный вопрос!», «Это очень важная тема!»

- Списки из 5 пунктов, где каждый начинается одинаково

*Для писем команде — эталон: как если бы руководитель писал сам, быстро, по делу, на живом русском языке. Если сомневаешься в тоне — спроси.*

# 16. Кросс-ролевые задачи, версионирование и первый запуск

## 16.1. Механика кросс-ролевых задач

Пример: «подготовь отчёт ФСИ» = бухгалтерия (расходы) + операционка (статус вех) + документ на выходе.

**Протокол:**

- Диспетчер определяет задействованные роли

- Последовательно вызывает каждого субагента с конкретным подзапросом

- Каждый субагент возвращает свой кусок

- Диспетчер собирает и формирует итоговый выход

- Self-check — на итоговом результате

## 16.2. Версионирование файлов

Конвенция: [тип]_[тема]_[YYYY-MM-DD]_v[N].[ext]

Пример: expense_log_2026-03_v1.md, contract_draft_kamayev_v2.docx

**Правило: перед обновлением файла — переименовать старую версию, добавив _v[N-1]. Не перезаписывать.**

## 16.3. Протокол создания нового скилла

- Диспетчер предлагает: «Создать скилл [название]?»

- Claude генерирует SKILL.md + вспомогательные файлы

- Показывает содержимое руководителю

- После подтверждения — сохраняет в .claude/skills/[name]/

- Прогоняет тестовую задачу с новым скиллом

## 16.4. Протокол обновления субагента

- Скопировать текущую версию в .claude/agents/backups/

- Внести изменения

- Прогнать золотой тест-кейс для этого субагента

- Тест пройден — оставить. Нет — откатить.

## 16.5. Сценарий первого запуска

**Пошагово — что делать в первый день:**

- Открыть Claude Desktop → вкладка Code → указать D:\RedPeak-HQ\

- Claude читает CLAUDE.md, память пуста → «Расскажите текущее состояние проекта»

- Рассказать: команда, договор, грант, сроки. Claude раскладывает на карточки.

- Первая задача: «Подготовь повестку созвона с Алексеем» → operations-agent + meeting-prep

- Вторая задача: «Таблица обязательств по договору с Камаевым» → legal-agent + contract-review

- Проверить: intake, тон, маркеры, стоп-точки, self-check

- Записать замечания в operations/calibration_log.md

# 20. Протокол обработки ошибок

| **Ситуация** | **Действие** |
| --- | --- |
| Некачественный результат | Указать конкретно. Скорректировать промпт субагента. |
| Галлюцинация | Пометить needs_review. Усилить правило в субагенте. Добавить в золотые кейсы. |

| Self-check пропущен | Усилить инструкцию в промпте субагента. Рассмотреть hook. |

# 21. Мониторинг стоимости

Правило в CLAUDE.md: «В конце каждой сессии — покажи текущий расход через /cost и предупреди, если он выше нормы.» Это автоматическое напоминание, не ручная проверка.

Первые 2 недели — дополнительно ежедневный контроль.

| **Показатель** | **Порог** | **Действие** |
| --- | --- | --- |
| Дневной расход | > 30% недельной нормы X5 | Оценить переход на X20 |
| Расход на субагента | > 50% общего расхода | Оптимизировать промпт |
| Простая задача | > 10K токенов | Проверить длину промпта |

# 22. Критерий успеха

- Задача в свободной форме → результат в нужном формате, без переключения окон.

- Память между сессиями: решения, обязательства, контекст сохраняются.

- Повторяющиеся задачи быстрее благодаря скиллам.

- Стоп-точки: 0 нарушений.

- Галлюцинации: 0 выдуманных фактов в финальных результатах (10 задач + 5 золотых кейсов).

- Self-check: каждый результат проверен перед выдачей.

- Метрики Фазы 5 достигнуты.

- Совпадение с эталоном в золотых кейсах > 80%.
