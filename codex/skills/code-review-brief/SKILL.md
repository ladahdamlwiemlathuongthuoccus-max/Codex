# code-review-brief

Сводка по изменениям в коде для нетехнического руководителя.

## Когда использовать

- "Что сделал разработчик за неделю?"
- "Покажи прогресс по коду"
- "Что изменилось в репозитории?"
- Подготовка к встрече с командой разработки

## Intake

1. Какой репозиторий? (путь или URL)
2. За какой период? (сегодня / неделя / с последнего обзора)
3. Чьи коммиты? (все / конкретный автор)
4. Уровень детализации? (краткий обзор / подробный)

## Шаги

1. Открыть репозиторий (git log за период)
2. Сгруппировать коммиты по смыслу (фичи / баги / рефакторинг / инфра)
3. Перевести на понятный язык
4. Оценить объём работ
5. Выделить ключевые изменения

## Формат результата

### Обзор разработки: [период]

**Автор(ы):** [имена]
**Коммитов:** [N]
**Файлов изменено:** [N]

#### Что сделано
- [Описание 1 — понятным языком, без технических деталей]
- [Описание 2]
- [Описание 3]

#### Ключевые изменения
| Что | Зачем | Статус |
|---|---|---|
| [Фича/исправление] | [Бизнес-причина] | готово / в процессе |

#### Метрики
- Строк добавлено: +[N]
- Строк удалено: -[N]
- Новые файлы: [N]

#### На что обратить внимание
[Если есть: незавершённые ветки, большие рефакторинги, потенциальные проблемы]

## Правила

- Переводить технические термины на бизнес-язык
- "Рефакторинг модуля авторизации" → "Улучшили внутреннюю организацию кода входа в систему"
- "Фикс race condition в API" → "Исправили ошибку, из-за которой запросы могли обрабатываться неправильно при высокой нагрузке"
- Не перегружать деталями. Руководителю нужен ответ на вопрос "движемся ли мы?"

## Запись в память

НЕ записывать. Сводка актуальна только на момент создания.
